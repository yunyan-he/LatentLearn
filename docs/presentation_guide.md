## 🛠️ 如何极速将 .md 文件导出为 PPTX / PDF 横版

我们采用的 **Marp (Markdown Presentation Ecosystem)** 是目前全球开发者最推崇的演示文稿编写方案。您可以通过以下两种最简单的方式将其转换为最终的 PPTX 或 PDF：

### 方案 A：使用 VS Code 插件（推荐，1分钟上手）
1. 在 VS Code 中，搜索并安装插件：**Marp for VS Code**。
2. 双击打开项目中的 [presentation_slides.md](file:///Users/aerin/Documents/LatentLearn/presentation_slides.md)。
3. 点击 VS Code 右上角的 **Marp 图标**（一个蓝色的小幻灯片图标），即可打开实时预览。
4. 点击预览窗口上方或编辑器标题栏的 **Marp 动作按钮**（或使用快捷键 `Cmd+P` 并输入 `Marp: Export Slide Deck...`）。
5. 在弹出的导出格式中选择 **PDF document** 或 **PowerPoint document (PPTX)**，即可一键生成完美的横版文件！

### 方案 B：使用命令行工具（CLI）
如果您熟悉终端，可以直接在项目根目录运行以下命令进行转换：
```bash
# 1. 全局安装 marp-cli
npm install -g @marp-team/marp-cli

# 2. 导出为 PDF 横版文件
marp --pdf presentation_slides.md -o LatentLearn_Presentation.pdf

# 3. 导出为 PowerPoint (PPTX) 文件
marp --pptx presentation_slides.md -o LatentLearn_Presentation.pptx
```

### Slide 2：The Cognitive Pain Point (线性陷阱)
* **视觉元素**：左右分栏，两张深灰卡片。左边是橙粉色调的“上下文污染”，右边是蓝青色调的“迷失主干”。
* **中文演说词**：
  > “在深入技术之前，我们先聊聊痛点。大家都用过 ChatGPT、Claude 或 Kimi。它们采用的是经典的‘单线程瀑布流’对话模式。
  >
  > 但对于 associative learners（关联型学习者），或者 ADHD（注意缺陷与多动障碍）等发散思维群体来说，这种线性模式是一个巨大的认知瓶颈。
  >
  > 比如在探索一个复杂话题时，我们常常会被 AI 回答里的某个生词或衍生技术吸引，想要深入挖掘。在传统聊天窗里，你只能不断追问。这带来了两大灾难：
  > 1. **上下文污染**：追问完细节后，模型的历史上下文中全是琐碎的分支杂音，再次回到主线时，AI 的回答质量显著下滑。
  > 2. **迷失主干**：当你满足了好奇心，想要回到最初的主线时，需要面临漫长的向上滚动，此时你的注意力已经涣散，学习心智被迫中断。这就是我们所说的‘瀑布流线性陷阱’。”

### Slide 5：Technical Architecture (技术架构层)
* **视觉元素**：简洁清晰的三层分布式系统架构图（Next.js 🚀 FastAPI ⚡ LangGraph），下方是三栏核心卡片。
* **中文演说词**：
  > “要支撑如此复杂的空间树状通信，底层的技术架构必须做到高内聚、低耦合。
  > 
  > 我们的系统采用了三层架构设计：
  > 1. **Next.js 前端**：作为高 reactive 的视图层，使用 App Router 与 TypeScript 5 严格类型约束，构建了完备的 Server-Sent Events 流式解析引擎和画布位置 Reducer，确保卡片平移与树状态更新的毫秒级同步。
  > 2. **FastAPI 异步 API 网关**：负责高并发长连接。我们在网关中加入了输入安全沙箱，彻底清洗潜在的 XSS 注入和提示词攻击。
  > 3. **LangGraph 状态机后端**：这是我们的大脑，采用有向无环图（DAG）和 Checkpointer 机制，对用户的多次探索进行分布式会话记忆追踪，并高效调用 OpenRouter 提供的顶级 LLM 服务。”

### Slide 6：LangGraph Agent Engine (智能体状态机)
* **视觉元素**：经典的 Mermaid 状态图预览，右边是 4 个关键节点的卡片（Router、Decomposer、Tutor、Offtopic Eval）。
* **中文演说词**：
  > “在这里，我们重点展示 LatentLearn 最硬核的 AI 模块——**LangGraph 智能体状态机**。
  > 
  > 传统的 AI 调用只是简单的单次 API 交互。而我们使用 LangGraph 将用户的单次输入编排成了一个高容错、可流转的 DAG。
  > 
  > 用户提交请求后：
  > 首先由 `intent_router` 识别意图，分流处理；
  > 如果用户提问过于宏大或模糊，`decomposer` 会将其拆解成结构化的子问题数组，避免一次性吐出过多内容压垮用户认知；
  > 随后 `anchor_locator` 在原文中计算相似度，高置信度地锁定引用区域，并将上下文送给 `tutor` 生成 Markdown 教学响应，同时 tutor 还会自适应提取简短的 `document_summary`；
  > 最后，`offtopic_eval` 登场，它会拿当前问答与 `document_summary` 进行语义相似度对比，精确判定这次讨论是否偏离了文档大纲。这几个 Agent 紧密协作，完成了单次交互的安全闭环。”

### Slide 7：Core Technical Breakthroughs (核心技术突破)
* **视觉元素**：红、蓝两块大对比色的卡片，展现混合流过滤、离题继承、早期退出。
* **中文演说词**：
  > “在开发过程中，我们攻克了多项业界公认的 AI 工程难点：
  > 
  > 第一个是**‘混合流过滤技术’**。在多智能体协同中，像拆解器（Decomposer）和树状态写入器（Tree Writer）输出的是结构化的 JSON 字符。如果直接和回答流混合，前端会直接播放难懂的 JSON 源码。我们通过深入 LangGraph 的底层事件驱动 API `astream_events`，实现了一套精准的节点白名单过滤机制，只有来自 `tutor` 节点的生成 token 才会被渲染，其他结构化状态则由专属的后台通道默默流转。
  > 
  > 第二个是**‘离题继承与熔断机制’**。大模型调用非常昂贵。如果用户在已经离题的卡片上继续无限深挖，我们无需每次都调用高成本的检测器。离题属性被设计为‘向下继承’，从而大幅度降低了 token 消耗。同时，在复合拆解（Decompose）模式下，一旦识别到多个子问题，系统便在 `tutor` 节点前触发‘早期退出’熔断，直接下发计划包，拒绝冗余执行，极致压榨每一毫秒的响应延迟。”

### Slide 9：智能学习核心链路与技术路线
* **视觉元素**：高对比毛玻璃表格，展示三大核心支柱：复合问题智能拆解、树状会话关联挂载、多维语义偏离检测。
* **中文演说词**：
  > “现在，让我们把视线聚焦到我设计的核心机制上——**智能学习生命周期链路与技术路线**。我们围绕非线性探索的痛点，构建了三大技术支柱，来全面接管用户在画布上的全生命周期轨迹：
  > 
  > 第一，**复合问题智能拆解**。当用户提出模糊或极其庞大的复杂问题时，后端的 decomposer 节点会在 tutor 节点响应前，通过 Early-Exit 早期熔断机制，首先将问题解构为有序的 QuestionPlan 计划包。这就避免了知识的一次性平铺，让用户能够沿着一条逻辑顺序，逐步击破疑惑。
  > 
  > 第二，**树状会话关联挂载**。当用户通过划词选中任何疑问点时，前端微菜单计算坐标映射，实时唤起 tree_writer 节点和 Context Anchor 原文定位器，在 Focus Tree 上挂载专属的局部卡片。这让所有的衍生探索都有据可依、绝不污染父卡片历史，彻底告别一维滚动杂乱。
  > 
  > 第三，**多维语义偏离检测**。在后台，offtopic_eval 节点时刻保持敏锐侦测。当用户的探索严重偏离文档大纲时，不仅其后续子卡片会自动继承离题标志以节约 LLM 开销，画布上更会精准弹出一键回归（Refocus）导流护栏，温柔地将发散的思路引导回思维主干，实现真正的高效学习。”
